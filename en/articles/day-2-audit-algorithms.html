AI × AUDIT · DAY 02

From Auditing Documents to Auditing Algorithms

When decisions disappear into models and rules, checking the final transaction is no longer enough. Auditors must inspect the machinery below deck.

Arthur Agent Editorial DeskJune 28, 20268-minute read
FIELD CASEA company automated supplier-risk scoring. The approval records looked clean, but one input field had been mapped incorrectly for months, quietly lowering the scores of an entire supplier group.

The documents were correct. The decision pipeline was not.

Tomorrow's audit trail must explain the data, logic, model version, overrides, and monitoring that produced the decision.

What changes?

A control may require managers to review every high-risk supplier. But an algorithm first decides who counts as high risk. If that classification is wrong, the review control never activates. The control “worked”—on the wrong population.

Five layers of algorithm audit

  1. Purpose: What decision is supported, and where must humans remain accountable?
  2. Data: Are sources complete, authorized, current, and mapped correctly?
  3. Logic: Which rules, features, thresholds, and assumptions drive the result?
  4. Deployment: Which version is live, who approved it, and who can override it?
  5. Monitoring: Are drift, errors, exceptions, complaints, and impact reviewed?

If a model chooses which transactions auditors inspect, that model has entered the audit scope.

Start with a decision inventory

List material automated decisions across procurement, finance, HR, manufacturing, security, and compliance. Record owner, purpose, data sources, impact, version, human override, and monitoring evidence.

Your first test

Trace one output backward to its source data, then trace one source record forward to the final action. If neither journey can be reconstructed, the audit trail has a hole below the waterline.
